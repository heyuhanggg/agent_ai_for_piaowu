# 历史故障案例复盘

## 案例1：订单服务OOM导致服务不可用

### 故障概述
- **时间**：高峰售票期间
- **影响**：order-service全部实例OOM崩溃，用户无法下单，持续约15分钟
- **级别**：P0

### 故障时间线
1. 热门演唱会开售，大量用户同时抢票下单
2. order-service收到大量并发创建订单请求
3. 订单创建逻辑中，一次性查询了用户所有历史订单用于校验（未分页）
4. 高并发下，每个请求都将大量数据加载到内存
5. JVM堆内存迅速耗尽，触发Full GC但无法回收，OOM崩溃

### 根因分析
- order-service在创建订单前，调用了 `orderMapper.selectByUserId(userId)` 查询用户历史订单
- 该查询未做分页限制，部分老用户有数千条历史订单
- 高并发下，多个线程同时加载大量数据，堆内存瞬间耗尽

### 解决方案
- **紧急**：重启服务，临时增大JVM堆内存到4G
- **修复**：将历史订单查询改为 `selectCountByUserId`，只查数量不查全量数据
- **防御**：
  - 所有数据库查询强制分页（框架层面拦截）
  - 配置 `-XX:+HeapDumpOnOutOfMemoryError` 便于事后分析
  - 增加JVM内存使用率告警（阈值85%）

## 案例2：Redis缓存雪崩导致数据库被打满

### 故障概述
- **时间**：缓存批量过期后
- **影响**：program-service响应时间从50ms飙升到5秒，大量请求超时
- **级别**：P1

### 故障时间线
1. 节目数据缓存统一设置了24小时过期
2. 凌晨批量导入节目数据时统一写入缓存
3. 第二天同一时刻，所有缓存同时过期
4. 大量请求穿透到数据库，数据库连接池耗尽
5. program-service响应超时，影响上游gateway-service

### 根因分析
- 所有节目缓存使用了相同的过期时间（24小时），没有添加随机偏移
- 缓存过期后没有互斥锁机制，导致大量请求同时回源数据库
- 数据库连接池配置为默认值（maxPoolSize=10），无法承受突发流量

### 解决方案
- **紧急**：手动刷新缓存，恢复数据库连接
- **修复**：
  - 缓存过期时间添加随机偏移（基础时间 + random(0, 3600)秒）
  - 引入互斥锁（Redis setnx），缓存失效时只允许一个请求回源
  - 增大数据库连接池（maxPoolSize=50）
- **防御**：
  - 热点数据使用永不过期策略，通过异步任务定期刷新
  - 添加本地缓存（Caffeine）作为二级缓存
  - 数据库连接池使用率添加告警

## 案例3：网关限流配置错误导致正常请求被拦截

### 故障概述
- **时间**：限流规则变更后
- **影响**：约30%的正常用户请求被限流拒绝，返回429状态码
- **级别**：P1

### 故障时间线
1. 运维人员调整gateway-service限流规则，将QPS阈值从1000调低至100
2. 配置变更后未经灰度验证直接全量生效
3. 当前实际QPS约300，大量请求被限流拦截
4. 用户反馈页面无法访问，客服收到大量投诉

### 根因分析
- 限流规则变更缺少审核机制，单人可直接修改生效
- 配置变更未做灰度发布，直接全量生效
- 限流阈值从1000直接降到100，降幅过大
- 缺少配置变更的自动告警机制

### 解决方案
- **紧急**：回滚限流配置到变更前的版本
- **修复**：
  - 配置变更增加审核流程（至少两人审核）
  - 限流规则变更支持灰度生效（先在一个实例生效，观察后再全量）
- **防御**：
  - 配置变更自动触发告警通知
  - 限流规则变更幅度超过50%时强制二次确认
  - 建立配置变更回滚快捷操作

## 案例4：支付回调超时导致订单状态不一致

### 故障概述
- **时间**：支付高峰期
- **影响**：部分用户支付成功但订单状态仍显示"待支付"
- **级别**：P1

### 故障时间线
1. 用户完成支付，第三方支付平台发起回调通知
2. pay-service接收回调，调用order-service更新订单状态
3. order-service处理缓慢（数据库连接池紧张），导致回调处理超时
4. 第三方支付平台认为回调失败，进行重试
5. 部分回调最终成功，部分始终超时，导致订单状态不一致

### 根因分析
- pay-service处理回调时同步调用order-service，未做异步化
- order-service在高峰期数据库连接池紧张，处理变慢
- 回调处理超时后，缺少补偿机制
- 没有定时对账任务来发现和修复不一致数据

### 解决方案
- **紧急**：手动查询支付平台订单状态，修复不一致的订单
- **修复**：
  - 支付回调改为异步处理（先返回成功，再异步更新订单）
  - 引入消息队列解耦pay-service和order-service
  - 增加支付回调幂等处理（基于支付流水号去重）
- **防御**：
  - 建立定时对账任务，每小时比对支付平台和本地订单状态
  - 添加订单状态不一致告警
  - pay-service处理回调超时时自动进入补偿队列
