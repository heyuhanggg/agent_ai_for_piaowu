# 常见故障排查手册

## NullPointerException（空指针异常）排查

### 问题现象
服务日志中出现大量 NullPointerException，导致请求处理失败，接口返回500错误。

### 排查步骤
1. 查看错误日志，定位异常堆栈中的具体代码行号
2. 检查该代码行涉及的对象是否可能为null
3. 查看上游调用链，确认入参是否合法
4. 检查数据库查询结果是否为空（未做null判断）
5. 检查Redis缓存是否返回null（缓存过期或key不存在）

### 常见原因
- 数据库查询结果未做空判断，直接调用方法
- Redis缓存过期或key拼写错误，返回null后未处理
- RPC调用返回的DTO对象中嵌套字段为null
- 集合操作时未判断集合是否为空
- JSON反序列化时字段缺失导致对象属性为null

### 解决方案
- **紧急处理**：定位具体代码行，添加null判断，紧急发布修复
- **长期优化**：
  - 在关键方法入口添加参数校验（@NotNull注解或手动检查）
  - 使用Optional包装可能为null的返回值
  - 数据库查询结果统一封装，返回空集合而非null
  - 代码审查中强制要求null安全处理

## 数据库连接池耗尽排查

### 问题现象
服务日志出现 "Cannot get a connection, pool error Waiting for connection timed out" 或 "HikariPool - Connection is not available"，大量请求超时。

### 排查步骤
1. 查看数据库连接池监控指标（active/idle/pending连接数）
2. 查看慢SQL日志，是否有长时间占用连接的SQL
3. 检查是否有未正确关闭的数据库连接（事务未提交/回滚）
4. 查看数据库服务器连接数是否达到上限
5. 检查最近是否有代码变更引入了连接泄漏

### 常见原因
- 慢SQL导致连接长时间占用，池中可用连接被耗尽
- 代码中手动获取Connection后未在finally中关闭
- 事务注解使用不当，长事务占用连接
- 连接池配置过小，无法支撑业务并发量
- 数据库服务器性能瓶颈（CPU/IO过高）

### 解决方案
- **紧急处理**：
  - 增大连接池maxPoolSize（临时扩容）
  - 杀掉数据库中的长时间运行SQL
  - 重启受影响的服务实例
- **长期优化**：
  - 优化慢SQL，添加合适的索引
  - 配置连接泄漏检测（leakDetectionThreshold）
  - 合理设置连接池参数（最大连接数、超时时间）
  - 引入读写分离，分散数据库压力

## Redis连接异常排查

### 问题现象
服务日志出现 "JedisConnectionException" 或 "RedisConnectionFailureException"，缓存操作失败，部分接口响应变慢或报错。

### 排查步骤
1. 检查Redis服务器是否正常运行（ping测试）
2. 查看Redis连接数是否达到maxclients上限
3. 查看Redis内存使用是否达到maxmemory上限
4. 检查网络连通性（服务实例到Redis之间）
5. 查看Redis慢日志（SLOWLOG GET）

### 常见原因
- Redis服务器宕机或正在进行主从切换
- Redis连接数耗尽（客户端连接池配置不合理）
- Redis内存使用达到上限，触发淘汰策略导致操作失败
- 网络抖动导致连接超时
- 大key操作阻塞Redis（如对百万元素的集合执行KEYS命令）

### 解决方案
- **紧急处理**：
  - 确认Redis服务状态，必要时重启Redis
  - 清理Redis中的过期大key
  - 检查并调大Redis的maxclients和maxmemory配置
- **长期优化**：
  - 合理配置客户端连接池参数（maxTotal、maxIdle、minIdle）
  - 避免使用KEYS、SMEMBERS等全量扫描命令，改用SCAN
  - 拆分大key，控制单个key的大小
  - 部署Redis哨兵或Cluster模式，实现高可用
  - 为缓存操作添加降级策略（缓存不可用时回源数据库）

## 接口响应超时排查

### 问题现象
用户反馈页面加载缓慢，接口响应时间超过预期（如超过3秒），部分请求直接超时返回504。

### 排查步骤
1. 查看接口响应时间监控指标（P99/P95/P50延迟）
2. 通过链路追踪查看请求在各服务间的耗时分布
3. 检查数据库查询是否存在慢SQL
4. 查看Redis操作是否有异常延迟
5. 检查JVM GC情况（是否频繁Full GC）
6. 检查下游RPC调用是否超时
7. 查看系统资源（CPU、内存、磁盘IO、网络带宽）

### 常见原因
- 慢SQL导致数据库查询耗时过长
- 下游服务响应慢，级联导致上游超时
- JVM频繁Full GC导致STW（Stop The World）
- 服务实例CPU或内存资源不足
- 网络带宽瓶颈或网络抖动
- 线程池队列积压，请求排队等待

### 解决方案
- **紧急处理**：
  - 扩容服务实例，分散请求压力
  - 开启限流降级，保护核心链路
  - 杀掉数据库中的慢查询
- **长期优化**：
  - 优化慢SQL，添加数据库索引
  - 引入缓存减少数据库访问
  - 设置合理的RPC超时时间和重试策略
  - 优化JVM内存参数，减少Full GC频率
  - 异步化非关键操作，缩短主链路耗时

## OOM（OutOfMemoryError）排查

### 问题现象
服务突然崩溃重启，日志中出现 "java.lang.OutOfMemoryError: Java heap space" 或 "GC overhead limit exceeded"。

### 排查步骤
1. 查看JVM堆内存使用趋势（是否持续增长不释放）
2. 获取heap dump文件进行分析（jmap -dump 或配置 -XX:+HeapDumpOnOutOfMemoryError）
3. 使用MAT（Memory Analyzer Tool）分析dump文件中的大对象
4. 检查最近的代码变更是否引入了内存泄漏
5. 查看GC日志，分析GC频率和回收效果

### 常见原因
- 大集合（List/Map）持续添加数据不释放
- 查询数据库时未分页，一次加载大量数据到内存
- 缓存没有设置过期策略，数据持续膨胀
- 线程局部变量（ThreadLocal）使用后未清理
- 第三方库的内存泄漏

### 解决方案
- **紧急处理**：
  - 重启服务恢复可用性
  - 临时增大JVM堆内存参数（-Xmx）
  - 如果是某个接口触发的，先限流或下线该接口
- **长期优化**：
  - 数据库查询必须分页，限制单次查询结果数量
  - 本地缓存使用Caffeine等带淘汰策略的缓存框架
  - ThreadLocal使用完毕后在finally中remove
  - 定期进行内存泄漏检测，配置告警
  - 配置合理的JVM参数：-Xms、-Xmx、GC策略
